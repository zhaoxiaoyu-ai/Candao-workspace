# Tappo App Style Kit v0.1 - 2026-06-08

## 文档定位

这份文档是 Tappo Phone App 端当前的风格系统草案，用于先统一产品气质、色彩、图标、插画、组件样张和页面套用规则，再进入页面塑造。

当前不要逐页做局部美化，也不要立即批量生成最终资产。先用这份 Style Kit 约束风格，确认后再生成小批量首批资产。

## 设计读法

Tappo Phone 是面向餐饮老板和员工的移动 WebView 产品。它不是传统后台，也不是纯营销 App。它应该是：

- 轻量、干净、智能感、运营感、可信赖。
- 2025-2026 当前移动 UI 气质，而不是旧式企业后台。
- 橙色主导品牌和动作，辅以清透柔和的智能氛围。
- 页面简约清晰，信息层级明确，不模板化，不“AI 味”。
- 在订单、支付、设置、报表等场景中保持专业和可重复使用的稳定感。

核心风格短语：

`橙色主导的清透软渐变智能运营 UI`

## 字体计划

- 主 UI 字体：`OPPO Sans 4.0` 优先，回退到 `OPPO Sans`、`-apple-system, BlinkMacSystemFont, "PingFang SC", "PingFang TC", sans-serif`。
- 数字指标字体：默认同 `OPPO Sans 4.0`；仅当数据卡需要更强数字识别时，可用 `Montserrat` 作为备选，并回退到 `OPPO Sans 4.0` 和系统 sans。
- 当前阶段不引入新的展示字体。
- 不使用负字距。
- 中文和港澳繁体文案必须允许换行，不能为了视觉整齐强制压缩。

## 当前阶段

当前阶段是 Style Kit 定义，不是最终资产生产。

本阶段要完成：

- 定义颜色 token。
- 定义渐变和阴影配方。
- 定义图标、插画和智能锚点规则。
- 定义组件样张。
- 定义页面套用矩阵。
- 定义反 AI 味、反模板化检查表。
- 定义首批资产生产门槛。

只有这份文档确认后，才进入首批资产生成。

## 不可妥协规则

- Tappo 橙色是主品牌色和主行动色。
- 蓝色只作为智能、系统、信息氛围辅助色，不能替代品牌主色。
- 小T不是固定吉祥物，而是可变的助手表达层。
- 详情页、支付页、点餐页、交易记录页必须克制。
- 智能感来自层级、解释、状态、建议和系统视觉，不来自泛泛的 AI 装饰。
- 原型内容是业务内容源，不能因视觉改版删除或弱化。
- 图标和插画必须统一风格，不能每页生成一套。
- 不允许生成假 logo、假界面文字、水印、棋盘格、随机英文文案。

## 风格比例

默认比例：

`70% 清晰运营 UI + 20% 橙色品牌温度 + 10% 柔和智能氛围`

按页面类型调整：

- 强智能感页面：智能氛围可提升到 `15-18%`。
- 中等智能辅助页面：智能氛围保持在 `8-12%`。
- 克制操作/详情页面：智能氛围控制在 `0-5%`。

## 页面智能感分层

### Tier 1：强智能感页面

适用页面：

- 首页封面。
- 角色选择。
- 老板端首页。
- 员工端首页 / 应用中心。
- 数据分析 / 报表。
- 应用管理。
- 设置总览。

允许使用：

- 柔和橙色主视觉。
- 小T助手或橙色智能光球。
- 清透渐变背景。
- AI 洞察卡。
- 运营建议卡。
- 高完成度图标和轻 3D 微插画。

限制：

- 每页只能有一个主智能视觉锚点。
- 智能视觉不能比主数据和主操作更抢眼。

### Tier 2：中等智能辅助页面

适用页面：

- 登录 / 注册。
- 员工 PIN 登录。
- 地区 / 组织选择。
- 点数购买 / 充值概览。
- 桌台列表。
- 预约列表。
- 支付设置。
- 设备绑定。

允许使用：

- 小型帮助提示。
- 智能状态 chip。
- 柔和图标卡。
- 简短说明性引导。

限制：

- 表单、列表和选择任务必须优先。
- 不使用大面积角色插画或大光球。

### Tier 3：克制操作/详情页面

适用页面：

- POS 点餐。
- 结账 / 订单确认。
- 支付成功。
- 交易记录 / 交易详情。
- 订单详情。
- 新增预约表单。
- 其他高频操作和审计类页面。

允许使用：

- 白色卡片。
- 清晰状态图标。
- 语义色状态。
- 对齐金额和订单信息。
- 明确底部操作。

禁止：

- 大面积 AI 插画。
- 小T 大头像。
- 智能光球主视觉。
- 装饰性渐变压过金额、订单、状态、操作。

## 用色纪律

- 一个页面只能有一个主色主导，Tappo App 中主导色必须是橙色。
- 蓝色只用于系统、信息、智能氛围，不用于抢主 CTA。
- 紫色只能作为极低频洞察辅助色，不能成为 AI 默认色。
- 成功、错误、预警必须语义化，不能为了好看随意替换。
- 图标 tile 底色不设固定清单；品牌色是橙色主导关系，不是每个 tile 都必须橙色，也不是只能使用下方几个参考配方。
- 图标 tile 可以按模块语义变化：支付/营收可偏橙或青，设备/连接可偏蓝或薄荷，预约/提醒可偏浅黄或蜜桃，员工/权限可偏蓝或浅紫，活动/引导可偏粉橙。
- 同一屏内建议控制在 `3-4` 个主要底色色系；除非有明确分组理由，不要让每个图标都像来自不同图标库。
- 新增图标底色色系必须保持清透、柔和、低饱和、轻渐变、统一光源、统一透明度和统一圆润质感。
- 不做整页橙色、整页蓝色、整页米色或整页紫色。
- 不使用高饱和多彩渐变做主视觉。

## 渐变配方

### 主按钮渐变

用于主 CTA、充值、确认、下一步。

```css
linear-gradient(180deg, #ffad6b 0%, #ff8a3d 52%, #f97316 100%)
```

### Tier 1 柔和品牌背景

用于封面、角色选择、老板首页顶部氛围。

```css
radial-gradient(circle at 78% 0%, rgba(255, 176, 111, 0.42), transparent 34%),
linear-gradient(180deg, #fff8f2 0%, #f7f8fb 100%)
```

### 智能浅蓝背景

用于 AI 洞察、系统健康、设置总览。

```css
linear-gradient(135deg, #fff8f2 0%, #eef7ff 52%, #f7fbff 100%)
```

### 橙色智能光球

只用于智能光球 / 系统中枢，不用于普通卡片。

```css
radial-gradient(circle at 38% 34%, #fff4dc 0%, #ffb06f 28%, #ff8a3d 52%, #f97316 76%, rgba(249, 115, 22, 0) 100%)
```

### 图标 tile 可变色原则

下方配方是色彩语法样例，不是可用底色列表。设计或生成资产时，可以围绕模块语义派生新的浅色渐变，只要它仍然符合 Tappo 的清透、柔和、低饱和、轻体积感和橙色品牌主导关系。

判断标准不是“是否用了指定颜色”，而是：

- 与同屏其他 tile 的明度、饱和度、光源一致。
- 与页面主色关系一致，不能抢走橙色 CTA 和关键操作。
- 与模块语义一致，不能为了丰富而随机换色。
- 与图形复杂度一致，不能有的偏扁平、有的偏重 3D、有的偏玻璃拟物。
- 新增明显色系时，需要记录适用模块和使用范围，避免后续扩散失控。

### 图标 tile 参考配方 - 品牌橙

参考起点：适用于品牌、收款、充值、关键入口方向。可按模块语义微调，不要求所有重要模块都使用橙色底。

```css
linear-gradient(135deg, #fff0e4 0%, #ffb06f 52%, #ffd9bd 100%)
```

### 图标 tile 参考配方 - 系统蓝

```css
linear-gradient(135deg, #eff7ff 0%, #b9dcff 50%, #d9f8ff 100%)
```

### 图标 tile 参考配方 - 健康薄荷

```css
linear-gradient(135deg, #dffbed 0%, #a8f3cf 48%, #eafcff 100%)
```

### 图标 tile 参考配方 - 奖励/引导

低频使用。

```css
linear-gradient(135deg, #ffe58a 0%, #ffb06f 48%, #ffd1e6 100%)
```

### 图标 tile 可变色边界

- 允许：浅橙、浅蓝、浅青、薄荷绿、浅黄、浅粉、浅紫的低饱和组合。
- 允许：同一 tile 内 2-3 个邻近色柔和过渡。
- 允许：根据模块语义选择色系，例如支付偏橙/青，设备偏蓝/薄荷，预约偏浅黄/橙，员工偏蓝/浅紫。
- 允许：为业务模块生成新的相邻浅色组合，但必须补充到评审样张或资产清单中。
- 禁止：高饱和彩虹渐变。
- 禁止：每个图标使用完全不同的明度、材质和光源。
- 禁止：大面积深色、脏色、灰黑渐变。
- 禁止：为了区分模块而牺牲整体调性。

## 阴影与材质

### 0 层：平面信息

用于卡片内行、详情页密集信息。

```css
box-shadow: none;
border: 1px solid rgba(9, 5, 2, 0.06);
```

### 1 层：轻卡片

用于操作页列表、交易记录、普通设置行。

```css
box-shadow: 0 4px 14px rgba(15, 23, 42, 0.04);
```

### 2 层：主卡片

用于首页、设置组、应用模块卡。

```css
box-shadow: 0 12px 28px rgba(15, 23, 42, 0.07), 0 2px 8px rgba(255, 138, 61, 0.05);
```

### 3 层：强智能视觉

仅用于 Tier 1 主视觉卡、智能光球、小T 浮层。

```css
box-shadow: 0 18px 44px rgba(15, 23, 42, 0.10), 0 8px 22px rgba(255, 138, 61, 0.12);
```

### 内高光

用于图标 tile、柔和半透明卡片。

```css
box-shadow: inset 0 1px 0 rgba(255,255,255,0.72), 0 10px 24px rgba(15,23,42,0.06);
```

阴影规则：

- 不用厚重黑影。
- Tier 3 页面只使用 0-1 层阴影。
- 一个卡片不要叠多个发光效果。
- 阴影服务于层级，不服务于炫技。

## 圆角、间距与布局

圆角：

- `8px`：小 chip、紧凑状态。
- `12px`：按钮、小图标 tile。
- `16px`：模块卡片、列表行、输入框、紧凑卡片的统一圆角。
- `20-24px`：仅用于少量主视觉、智能主卡或大面积插画容器，不作为模块卡片默认圆角。
- `999px`：胶囊、分段控件、浮动按钮。

间距：

- 页面左右边距：`16px`。
- 390px 设计稿中主视觉可用 `16px` 对齐。
- 区块间距：`16px`。
- 卡片内边距：`16-24px`。
- 密集行内边距：`12-16px`。
- 图标与文字间距：`8-12px`。

布局：

- App 设计目标宽度：`390px`。
- 当前阶段先忽略 `402px` 历史宽度；页面塑造和样张评审以 `390px` 为准。
- 手机壳最大宽度：`440px`。
- 禁止卡片套卡片。需要层级时，用分割线、浅底区域或列表行。

## 字号与信息层级

| 角色 | 字号 | 字重 | 行高 | 用途 |
| --- | --- | --- | --- | --- |

| 主视觉标题 | `24-28px` | 700 | `32-36px` | 封面和 Tier 1 页面 |
| 区块标题 | `16px` | 650/700 | `24-28px` | 卡片组标题 |
| 卡片标题 | `15-16px` | 650/700 | `22-24px` | 模块卡标题 |
| 正文 | `14px` | 400/500 | `20-22px` | 主文案、列表行 |
| 辅助信息 | `12px` | 400/500 | `16-18px` | 时间、标签、说明 |
| 底部导航 | `11-12px` | 500 | `15-16px` | 仅底部导航 |

规则：

- 正文不得低于 `14px`。
- 辅助信息不得低于 `12px`。
- 大数字一张卡只突出一个。
- 金额、订单数、状态必须对齐清楚。
- 不能用小字堆信息来制造“高级感”。

## 图标系统

### 系统线性图标

用于：

- 底部导航。
- 顶部操作。
- 返回、搜索、通知、设置、箭头。
- 交易记录、详情页、操作页行内图标。

规则：

- 尺寸：`20-24px`。
- 线宽：`1.8px`。
- 圆角端点和圆角连接。
- 继续沿用 `TpLineIcon` 的代码原生 SVG 方向。
- 不混用多套线性图标风格。

### 柔和模块图标 tile

用于：

- 应用中心。
- 设置入口。
- 数据分析模块。
- 角色选择。
- 员工工作台。

规则：

- tile 尺寸：`48-64px`。
- tile 圆角：默认 `16px`。
- 背景颜色可按模块语义变化，不固定为某几个颜色，也不要求每个模块套用同一底色。
- 必须沿用统一的清透软渐变语法，保持光源、透明度、圆角和体积感一致。
- 同一屏内控制在 `3-4` 个主要底色色系，避免图标像临时拼贴。
- 图形居中，轮廓清晰，低细节。
- 不能嵌入文字。
- 光源统一为左上或上方。

### 状态图标

用于：

- 交易卡。
- 支付状态。
- 设备健康。
- 桌台状态。
- 预约状态。

规则：

- 尺寸：`32-40px`。
- 颜色必须语义化。
- 状态不能只靠颜色表达，必须配文字或图标。
- Tier 3 页面不能使用小T或大插画替代状态。

## 插画系统

允许三类资产：

- 小T / 助手表达。
- 橙色智能光球 / 系统中枢。
- 模块微插画。

### 小T / 助手表达

适用：

- 封面。
- 角色选择。
- 老板首页。
- 员工首页。
- 空状态。
- 帮助引导。

规则：

- 小T可以变化姿态、裁切、表情、头像形态。
- 小T不是固定吉祥物，不需要每页出现。
- 小T不能替代 Tappo logo。
- 小T不能压过数据和主操作。
- 不做幼稚玩具感，保持友好但专业。

### 橙色智能光球 / 系统中枢

适用：

- 数据分析。
- 设置。
- 系统健康。
- AI 洞察。
- 应用管理。
- 用量预测。

规则：

- 橙色是核心。
- 蓝/青只作为边缘反光。
- 造型应克制、精确、可信赖。
- 不能变成普通渐变 blob。
- 不能做暗黑赛博或霓虹 AI。

### 模块微插画

适用：

- POS / 设备设置。
- 支付配置。
- 预约引导。
- 员工权限。
- 应用开通。

- 轻 3D / 半扁平混合。
- 圆润几何，低细节。
- 柔和投影，清透渐变。
- 不写文字、不放假 logo、不做复杂场景。

## 组件样张规则

### 1. 封面主视觉

结构：

- 官方 logo。
- 产品名 / 简短定位。
- 一个小T或智能光球主视觉。
- 一个橙色主 CTA。
- 可选次级入口。

要求：

- 清透、少元素、无海报化。
- 不使用泛 AI 聊天机器人模板。

### 2. 角色选择卡

结构：

- 老板角色卡。
- 员工角色卡。
- 每张卡有角色图标、说明、主要能力。
- 选中或推荐路径用橙色强调。

要求：

- 两个角色要有不同业务气质。
- 不做两个完全相同的模板卡。

### 3. AI 洞察卡

结构：

- 洞察标签。
- 明确发现。
- 原因解释。
- 建议动作。
- 可选置信度或数据来源。

要求：

- 用“辅助判断”的语气，不用“自动决策”的语气。
- 不使用“AI magic”“一键智能”等空泛文案。

### 4. 数据指标卡

结构：

- 一个主数字。
- 两个以内次级指标。
- 状态 chip。
- 可选进度条或小图表。

要求：

- 数字优先，图表辅助。
- 金额和状态必须对齐清楚。

### 5. 应用模块卡

结构：

- 柔和模块图标 tile。
- 模块名称。
- 一句运营说明。
- 状态 / 开通 chip。
- 主操作按钮。

要求：

- 图标风格必须统一。
- 不做促销 banner 堆叠。

### 6. 设置分组

结构：

- 分组标题。
- 健康/连接状态 chip。
- 设置行。
- 可选系统中枢或设备微插画。

要求：

- 设置页可以有智能感，但不能像营销页。
- 重要状态必须可扫读。

### 7. 操作详情卡

结构：

- 白色卡片。
- 标签和值。
- 状态图标。
- 必要时底部固定操作。

要求：

- 不用大插画。
- 不用装饰渐变盖住金额或订单信息。
- 原型字段必须保留。

### 8. 底部导航

结构：

- 白色或轻磨砂底。
- 线性图标。
- 文案 `11-12px`。
- 激活态使用橙色图标/文字和浅橙底。

要求：

- 不使用生成图标。
- 不混用线性和面性图标。
- 不随意加入大中间按钮，除非页面明确需要。

## 去 AI 味检查表

如果出现以下情况，需要重做：

- 页面像通用 AI SaaS 模板。
- 主色变成蓝紫渐变。
- 到处都是光球、星光、机器人头。
- 文字出现“AI magic”“智能赋能”“一键生成一切”等空泛表达。
- 智能视觉没有解释、建议、总结或状态作用。
- 每个页面都重复同一套 hero。
- 替换 logo 后可以变成任意产品。

Tappo 自检：

- 橙色是否明确拥有品牌和主操作？
- 页面是否仍像餐饮运营产品？
- 金额、订单、预约、设备、支付状态是否一眼可读？
- 智能感是否真的帮助用户理解或行动？
- 原型内容是否完整保留？

## 去模板化检查表

拒绝以下模式：

- 顶部大渐变 + 中间三张等宽卡 + 底部按钮的通用模板。
- 所有页面都用同一种卡片堆叠。
- 所有图标都是随机渐变方块。
- 所有标题都写得像营销页。
- 数据卡只有视觉没有业务优先级。
- 详情页为了“高级”牺牲可读性。

修正方向：

- 先确定页面任务，再决定视觉强度。
- 每个页面只保留一个主视觉锚点。
- 强化业务对象：门店、订单、预约、支付、设备、员工、应用。
- 用一致的 token 和组件，而不是每页重新发明样式。

## 可访问性与信任检查

- 正文至少 `14px`。
- 辅助信息至少 `12px`。
- 主按钮高度至少 `44px`。
- 触控目标至少 `40px`，优先 `44px`。
- 状态不能只靠颜色表达。
- 操作页不能让装饰排在任务前面。
- 金额、订单号、支付方式、状态必须高对比可读。
